| 2026-07-15 | L1 | Port `CommandCreoleStyle`'s observable match semantics directly (hand-rolled creole/legacy/legacyEol matchers in `command/CommandCreoleStyle.ts`) instead of porting `com.plantuml.ubrex` (the ~30-file bracketed-regex DSL upstream's real `CommandCreoleStyle` is built on). | `ubrex` is a general-purpose pattern-matching ENGINE, not creole-specific logic — same category as graphviz-ts (a foundational dependency, out of mission charter). Verified the three challenge classes (`ChallengeOneOrMoreUpToOldVersion`, `ChallengeUpTo`, `ChallengeOneOrMore`) produce exactly "creole: lazy 1+ up to next token", "legacy: lazy 0+ up to close tag", "legacyEol: greedy 1+ to end of line" — reproduced those three semantics directly against `String`/`RegExp`, not the DSL. | none — mission brief explicitly authorized this trade-off ("the pattern architecture matters more than tag count") |
| 2026-07-15 | L1 | `leaf-sizing.ts` (the DOT-layout node-sizing path, `measureLeafNode`/`maxLineWidth`/`atomHeightBonus`) is left UNTOUCHED — still measures raw markup text (e.g. `**bold**` counts all 8 chars), not creole-style-aware. Only `EntityImageDescriptionSupport.ts#buildTextBlock` (the RENDER path) was cut over. | `layout.ts` calls `measureLeafNode` exclusively for DOT node sizing (grep-verified — `EntityImageDescription`/`buildTextBlock` is never reached from the layout path, only from `renderer.ts`). Leaving `leaf-sizing.ts` untouched makes the DOT gate PROVABLY unaffected by this iteration (byte-identical inputs to that file), the safest possible choice under "DOT gate FROZEN". Reconciling box/actor/usecase/note sizing with the new atom model (bridging `FontSpec`<->`FontConfiguration`, stripped-markup width, heading-font height growth) is real, separate work — deferred to a follow-up once corpus reach is measured (candidate: paired with L3's "full re-measure" pass). Residual: an entity with creole style markup in its display may render slightly narrower than its layout-computed box (cosmetic only, not a new class of bug — no acceptance fixture currently exercises this). | ledgered — see `plans/e2r-creole/ledger.md` |
| 2026-07-15 | L1 | `manageSpecialChars` (`<U+NNNN>`/`&#NNN;` escape resolution, upstream `legacy/AtomText.java`) is NOT re-ported inside the new `AtomText`-equivalent text-run construction. | Already fixed at a HIGHER layer (G1 I4c mechanism 1's `resolveTextEscapes`, `parse-helpers.ts#finalizeDisplay`), which runs on the WHOLE `node.display` string before `buildTextBlock` ever splits it into lines/atoms — by the time the creole pipeline sees a line, no `<U+XXXX>`/`&#NNN;` pattern remains for a per-atom pass to find. Re-porting it here would be a no-op, adding pure duplication risk (two independent copies of the same resolution logic silently drifting) for zero behavior change. | none |
| 2026-07-15 | L1 | Bullet lists (`StripeStyleType.LIST_WITHOUT_NUMBER`/`LIST_WITH_NUMBER`, upstream's `*`/`#`-prefixed lines, FULL-mode-only patterns in `CreoleStripeSimpleParser`) are not classified or rendered at all — `classifyStripeLine` never produces these values. | Not in L1's acceptance surface (no named fixture uses bullet lists inside a descdiagram entity display); implementing bullet-glyph rendering (`Bullet`/`AtomTextUtils.createListNumber`) is a separate, self-contained feature with zero interaction with the style-run/heading work this iteration targets. | ledgered — `StripeStyleType.ts`'s doc comment |
| 2026-07-15 | L1 | `<u:color>`/`<w:color>`/`<s:color>` (extended-color-suffixed activation patterns) are not recognized at all — `CommandCreoleStyle.ts`'s activation regex sources match only the bare `<u>`/`<w>`/`<s>` forms. | Mission brief NOT-in-scope list names `<u:>` explicitly (L2). Recognizing the tag boundary while ignoring the captured color would be a new partial-support behavior change with zero acceptance-fixture benefit and non-zero regression risk; left exactly as unrecognized as pre-L1 (falls through to literal text, unchanged). | none |
