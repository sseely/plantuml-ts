| 2026-07-15 | K1 | Extended `"transparent"`/`"background"` keyword collapse (upstream `HColorSet#parseColor` java:82-83, case-insensitive, both -> `HColors.none()` -> canonical `#00000000`) to EVERY fill/stroke via `resolveColorToSvgHex`, not just `paint.ts#isTransparentColor`'s existing backcolor-only gate | The existing `isTransparentColor` gate (G1 I5d) only covered `setupBackcolor`'s own already-fixed instance of this gap; `fixColor` (fill/stroke, not just backcolor) had the SAME literal-string-equality bug (`color === '#00000000'` compared the RAW string, so a named `"transparent"` fill/stroke never collapsed to `fill="none"`). Same root cause, same fix, generalized to its true reach — not a new mechanism | — |
| 2026-07-15 | K1 | On an unresolvable token (not a keyword, not valid hex, not a registered name), `resolveColorToSvgHex` returns the input UNCHANGED — no WHITE (`#FFFFFF`) fallback, unlike upstream's `getColorOrWhite` | Upstream's WHITE fallback runs at the SkinParam/`ColorParser` parse BOUNDARY, before an `HColor` ever reaches `SvgGraphics` — by the time upstream's `SvgGraphics` draws, every color is already a resolved, valid `HColor`. This port's `Paint` design defers resolution to the SVG-EMISSION layer instead ("stored verbatim, interpreted late" — `paint.ts`'s own pre-existing doc comment). Applying a WHITE fallback at THIS layer would risk silently clobbering non-color strings not yet audited at this choke point (`url(#id)` gradient references, any literal this port hasn't accounted for) with no way to distinguish "genuinely invalid color" from "not meant to be resolved here" — an unbounded blast radius for a narrowly-scoped iteration. Verified via the full 355-fixture before/after per-fixture diff-count scan: 0 regressions, 53 improvements, confirming the conservative choice didn't leave any in-scope fixture unfixed | — |
| 2026-07-15 | K1 | Gradient dedup key in `paintToSvg`/`createSvgGradient` now built from RESOLVED stop values, not raw input strings | Matches upstream's own dedup semantics (`Map<List<Object=HColor>, String>`, keyed on resolved `HColor`/`XColor` RGBA equality, not on original spelling) — a differently-spelled-but-equal-color gradient (`red\|#FF0000` vs `#FF0000\|#FF0000`) now dedups to the same id, matching the jar. Both id schemes (`paint.ts`'s content-hash and `svg-graphics-core.ts`'s seed+counter) are this port's own invention, not jar-matched literal ids (D2'/D3), so changing the hash BASIS changes no oracle-compared output — verified via the full test suite (0 regressions) | — |
| 2026-07-15 | K1 | Pre-existing tests asserting raw/unresolved color strings (`fill="red"`, `fill="cornsilk"`, `#LightBlue`, etc.) updated to their jar-verified canonical hex, rather than left broken or the fix reverted | Those assertions were pinning the BUG (raw pass-through) as correct behavior — the natural, intended consequence of closing T19/I2. 30 tests across 11 files updated; every replacement hex value either jar-verified directly against a cached golden SVG or computed from the verbatim-ported `ColorTrieNode` table (itself cross-checked line-for-line against the Java source) | — |
